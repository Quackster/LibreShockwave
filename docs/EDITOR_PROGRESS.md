# Goal: Port the Java Editor to C++ Qt

This repository's C++ editor direction is to replace the current GTK/libadwaita editor prototype with a C++ Qt editor that ports the Java editor feature set from:

`/opt/git/LibreShockwaveJava2/editor/src/main/java/com/libreshockwave/editor/`

Keep this file updated as the migration progresses. When a feature is implemented, partially implemented, intentionally deferred, or removed from scope, update the relevant checklist and notes here in the same change.

## Source of Truth

Use the Java editor in `/opt/git/LibreShockwaveJava2/` as the behavioral and UI reference, especially these packages:

- `com.libreshockwave.editor`
- `com.libreshockwave.editor.panel`
- `com.libreshockwave.editor.docking`
- `com.libreshockwave.editor.cast`
- `com.libreshockwave.editor.score`
- `com.libreshockwave.editor.script`
- `com.libreshockwave.editor.debug.ui`
- `com.libreshockwave.editor.preview`
- `com.libreshockwave.editor.property`
- `com.libreshockwave.editor.extraction`

The Qt implementation should use the existing C++ SDK/player/runtime code in this repository rather than embedding or depending on the Java project.

## Required Direction

- Remove the current GTK/libadwaita C++ editor project under `cpp/apps/editor`.
- Remove the `libreshockwave_editor` GTK/libadwaita build target and package checks from `cpp/CMakeLists.txt`.
- Add a new Qt-based C++ editor target, keeping the executable name `libreshockwave_editor` unless there is a strong build-system reason to rename it.
- Prefer Qt Widgets for the desktop-style Director editor UI.
- Use Qt docking primitives where they fit, but preserve the Java editor's customizable panel behavior and layout persistence.
- Keep native player/editor dependencies separate. Do not make the existing GTK native player migration part of this goal unless it becomes necessary.

## Core UI Scope

Port the Java editor's main application structure:

- Main window equivalent to `EditorFrame`.
- Shared editor state equivalent to `EditorContext`.
- Menu bar equivalent to `EditorMenuBar`.
- Toolbar equivalent to `EditorToolBar`.
- Open/close Director file flow with recent directory/project preference persistence.
- Start screen and external parameters dialogs where present in Java.
- Default Director MX 2004-style workspace layout.
- Window title updates based on the opened movie/cast file.

## Customizable Panels And Docking

The Qt editor must support the Java editor's customizable panel workflow:

- Floating panels.
- Docked panels.
- Tabbed dock groups.
- Split dock zones.
- Dock to left, right, top, bottom, and center areas.
- Drag/snap docking behavior or a Qt-native equivalent that gives the same user control.
- Right-click/context actions for docking, undocking, splitting, and closing panels.
- Persistent layout save/load equivalent to `LayoutPersistence`.
- Reset layout action.
- Stable panel identity by panel id, not by translated/display title.

Default visible panels:

- `stage` - Stage
- `score` - Score
- `cast` - Cast
- `property-inspector` - Property Inspector
- `script` - Script
- `message` - Message
- `tool-palette` - Tool Palette

Default hidden or secondary panels:

- `paint` - Paint
- `vector-shape` - Vector Shape
- `text` - Text
- `field` - Field
- `sound` - Sound
- `color-palettes` - Color Palettes
- `bytecode-debugger` - Bytecode Debugger
- Detailed Stack View debug window

Port each Java panel into a Qt equivalent:

- Stage window: movie rendering, frame display, stage sizing/background, file-aware title.
- Score window: custom score grid, channel headers, marker bar, playback head, frame/channel data model.
- Cast window: list/grid modes, thumbnails, search, selection sync, context menu, cast member opening.
- Property Inspector: sprite, member, behavior, and movie tabs.
- Script editor: script/context selectors, Lingo source display/editing surface, syntax highlighting, handler navigation.
- Message window: output area and command input.
- Tool Palette: Director-style tools.
- Paint window: bitmap editing shell and selected bitmap loading.
- Vector Shape window: vector shape editing shell.
- Text editor window: rich/styled text display/editing for text members.
- Field editor window: field text editing.
- Sound window: sound metadata and playback controls.
- Color Palettes window: palette listing/preview.
- Bytecode Debugger window: script browser, bytecode list, state inspection tabs, watches, debug toolbar, keyboard shortcuts.
- Detailed Stack View: call stack, operand stack, arguments, receiver/object details.

## Menus And Commands

Recreate the Java editor menu structure in Qt:

- File: new movie/cast stubs, open, close, save stubs, import/export stubs, preferences, exit.
- Edit: undo/redo, cut/copy/paste/clear/select all, find/replace stubs, sprite editing stubs.
- View: zoom, sprite overlays, sprite toolbar, keyframes, grids, guides.
- Insert: keyframe, marker, remove frame, media element entries.
- Modify: movie, sprite, cast member, frame, font, paragraph, external parameters.
- Control: play, stop, rewind, step forward/backward, loop playback.
- Debug: step into/over/out, continue, breakpoints, detailed stack window, trace handler.
- Window: all panel visibility toggles and reset layout.
- Help: about dialog.

Stubbed Java actions may remain disabled initially in Qt, but they should be present so the UI surface matches the Java editor and can be filled in incrementally.

## Data, Preview, And Editing Behavior

Port the supporting Java editor behavior where it exists:

- Selection manager and selection events.
- File scanning/member resolution.
- Cast member metadata model.
- Bitmap key/extraction task models.
- Generic, palette, script, sound, and text previews.
- Palette descriptions and channel names.
- Instruction formatting.
- Asset export flow currently exposed by `AssetExportCli`.
- Audio playback controller/editor audio backend behavior, using native C++/Qt audio APIs where appropriate.
- Preferences persistence for recent projects, last open directory, and user settings.

## C++/Qt Architecture Notes

Suggested C++ structure:

- `cpp/apps/editor-qt/` or replacement `cpp/apps/editor/` after removing GTK code.
- `MainWindow` for the Qt main window.
- `EditorContext` as a QObject-based shared state and signal source.
- One QWidget/QDockWidget-backed class per editor panel.
- Separate model classes for score, cast, selection, preview, and debug data.
- Use existing C++ parsing/rendering/debug classes from `cpp/src` and public headers.
- Keep UI code thin; put reusable editor logic in testable non-widget classes where practical.
